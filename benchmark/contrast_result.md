## Methodology

### Naive Approach

- Rebuilds the system prompt on **every** call by injecting the current timestamp and a random session ID.
- Tool schemas are re-serialized with a new `request_id` (UUID) each turn.
- No prompt layering: system + context + user are concatenated in an unstable order.
- Result: DeepSeek's prefix cache never finds a matching prefix → ~0% hit rate.

### Optimized Approach (llm-cache-optimizer)

- Uses `CacheAwareClient` with strict 6-layer ordering:
  `core_system → tool_schema → static_context → session_memory → history → runtime`
- Layers 1–3 are **stable**: they never change across turns within a session.
- `CanonicalSerializer` normalizes timestamps, UUIDs, and request IDs.
- `SessionMemory` compresses growing history into a compact summary every few turns.
- Result: DeepSeek matches the stable prefix → high hit rate (typically >80% after turn 1).
